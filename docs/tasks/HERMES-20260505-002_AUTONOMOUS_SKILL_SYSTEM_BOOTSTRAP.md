## 1. Objective

Сделать из Гермеса не просто исполнителя отдельных команд, а сильного проектного агента на вырост: чтобы он быстро входил в контекст, сам создавал и улучшал свои рабочие навыки, выполнял задачи эффективнее с каждым циклом, но не нарушал безопасность, приватность, медицинско-юридические границы и архитектуру проекта.

Результат этой задачи - не “один ответ”, а рабочая система для Гермеса:

1. Гермес сразу входит в контекст проекта Olga Wellness AI.
2. Гермес понимает свою роль, права, ограничения и зоны риска.
3. Гермес составляет карту своих возможностей и недостающих навыков.
4. Гермес проектирует и создает первичный набор собственных навыков/скиллов в безопасном формате.
5. Гермес вводит цикл самоулучшения: `задача -> выполнение -> проверка -> выводы -> обновление навыка -> отчет`.
6. Гермес формирует правила, которые потом можно переносить на другие проекты.

Важно: “самообучение” здесь означает не изменение весов модели, а создание внешней рабочей памяти: навыков, шаблонов, чек-листов, тестов, runbook'ов и отчетов.

## 3. North Star

Проект Olga Wellness AI должен стать безопасным Telegram-first сервисом нутрициологической навигации:

`анкета -> ручная оплата -> сбор данных/анализов -> AI-черновик -> внутренний критик -> human review -> клиентское досье -> сопровождение`

Не пересказывай все документы. Выдели то, что нужно для действий.

### Phase 2. Инвентаризация собственных возможностей

Определи, какие типы задач ты можешь выполнять лучше всего:

- аудит стратегии;
- аудит бота;
- аудит клиентского пути;
- русский UX/copywriting;
- safety review;
- growth review;
- анализ кода;
- составление task packets;
- документация;
- подготовка тест-кейсов;
- контроль качества;
- подготовка к переносу на VPS/домен/mini app.

Для каждой категории укажи:

- какую пользу даешь;
- какие входные данные нужны;
- какой результат выдаешь;
- какие риски;
- нужна ли проверка человеком.

### Phase 3. Создание собственной системы навыков

Создай папку:

`docs/hermes_skills/`

Создай индекс:

`docs/hermes_skills/index.md`

Создай первичный набор навыков в виде коротких, практичных `.md` файлов:

1. `project-context-loader.md` - как быстро входить в контекст проекта.
2. `task-packet-executor.md` - как брать задачу и доводить до отчета.
3. `wellness-safety-auditor.md` - как проверять медицинско-юридические риски.
4. `premium-russian-copy-editor.md` - как убирать шаблонность, ошибки и англицизмы.
5. `telegram-bot-flow-auditor.md` - как проверять клиентский путь бота.
6. `growth-and-offer-critic.md` - как проверять коммерческую ценность и упаковку продукта.
7. `knowledge-sync-reporter.md` - как фиксировать результаты в GitHub/Notion/Obsidian без секретов.
8. `launch-readiness-checker.md` - как проверять готовность к pilot/public launch.
9. `lab-data-safety-reviewer.md` - как проверять работу с анализами без риска неверных выводов.
10. `agent-self-improvement-loop.md` - как после каждой задачи улучшать свои навыки.

Каждый навык должен быть коротким и прикладным:

- когда использовать;
- входные данные;
- пошаговый workflow;
- критерии качества;
- что запрещено;
- формат результата.

Не создавай огромные трактаты. Навык должен помогать действовать, а не превращаться в архив.

### Phase 4. Система самоулучшения

Создай правило, которое будет применяться после каждой задачи:

1. Что в задаче повторяемо?
2. Можно ли превратить это в навык/чек-лист/шаблон?
3. Что было ошибкой или риском?
4. Как проверить это в следующий раз?
5. Нужно ли обновить существующий навык?
6. Нужно ли создать новый навык?
7. Что записать в отчет, чтобы следующая модель стала умнее?

Запиши это в:

`docs/hermes_skills/agent-self-improvement-loop.md`

### Phase 5. Лестница автономности

Создай в отчете лестницу доступа Гермеса:

- Level 0: read-only аудит.
- Level 1: документация и task packets.
- Level 2: тексты, UX, тест-кейсы.
- Level 3: ограниченные правки кода по конкретной задаче.
- Level 4: запуск проверок/смоуков по разрешенным командам.
- Level 5: production/VPS/payment/client-facing actions только после отдельного approval.

Для каждого уровня определи:

- что можно;
- что нельзя;
- какой отчет обязателен;
- когда нужен Codex/Olga approval.
