# Единый центр разработки проекта Olga Wellness AI

Дата обновления: 2026-05-01
Статус: active source of truth for GitHub
Проект: Telegram-first нутрициологическая навигация Ольги Зинченко
Репозиторий: `olyalyazinchenk-wq/Zinchenko_wellness_al`

## 1. Зачем существует этот документ

Это единое поле разработки проекта для GitHub. Сюда сведена безопасная проектная информация из локальных документов: стратегия, продуктовая линейка, текущий этап, архитектура, правила безопасности, блокеры, план запуска и ссылки на опорные документы.

Этот документ можно давать новому разработчику, аудитору, модели Antigravity/DeepSeek или помощнику, чтобы он быстро вошёл в контекст без доступа к `.env`, токенам, клиентским анализам, PDF, фото, runtime-файлам и персональным данным.

## 2. Главная цель

Мы строим не “бот ради бота”, а управляемый Telegram-сервис:

`анкета -> ручная оплата -> AI-черновик -> экспертная проверка -> клиентское досье -> сопровождение`

Клиент должен получить не диагноз и не лечение, а премиальную нутрициологическую навигацию:

- сбор жалоб, питания, сна, стресса, ЖКТ, желчеоттока, образа жизни и фоновых факторов;
- анализ имеющихся данных и документов;
- гипотезы по дефицитным рискам и перегружающим факторам;
- список анализов и обследований, которые стоит обсудить/сдать;
- вопросы к врачу при красных флагах;
- план действий на ближайшие дни, недели и месяцы;
- нутрицевтические ориентиры в рамках безопасности, без формата лечения.

## 3. Текущий этап

Этап: `controlled concierge pilot`.

Публичный массовый запуск пока заблокирован. Сейчас задача не расширять бесконечно продукт, а довести один безопасный платный клиентский цикл до конца.

Текущий целевой цикл:

1. Клиент заходит в Telegram-бот.
2. Видит понятную витрину продуктов и пример результата.
3. Выбирает продукт.
4. Проходит расширенную анкету.
5. Отправляет анализы, фото, PDF, выписки, голосовые или пишет данные текстом.
6. Если анализов нет, получает маршрут сдачи через HelloDoc.
7. Оплачивает вручную.
8. Оператор подтверждает оплату.
9. Система готовит AI-черновик.
10. Черновик проходит внутреннего критика и human review.
11. Только после проверки клиент получает результат.
12. В течение периода продукта клиент может досылать вопросы и документы.

## 4. Продуктовая линейка

Демо-пример результата: 0 ₽ на этапе пилота.

Назначение: показать качество, структуру и отличие от бесплатных советов из интернета. Не является полным персональным разбором.

Разбор на 7 дней: 3 900 ₽.

Назначение: первая ясность и безопасный старт. Подходит, когда клиент хочет понять, с чего начать, какие анализы нужны, какие первые шаги сделать.

Персональный разбор на 30 дней: 6 900 ₽.

Назначение: основной флагман. Полная анкета, работа с анализами/документами, гипотезы, список анализов, план на 3 дня / 2 недели / 1 месяц / 1-3 месяца, вопросы к врачу, нутрицевтические ориентиры, возможность досылать вопросы и документы в течение месяца.

VIP-сопровождение на 30 дней: 14 900 ₽.

Назначение: более плотное ведение, приоритетная ручная связь, корректировки по реакции организма, обработка дополнительных документов и вопросов в течение месяца. VIP не является медицинским лечением.

## 5. Позиционирование

Ключевая ценность:

`Система, которая собирает картину целиком, отделяет факты от гипотез, показывает безопасный следующий шаг, помогает подготовиться к врачу и даёт нутрициологический маршрут без самоуверенной медицины.`

Клиентская боль:

- непонятно, что происходит с самочувствием;
- анализы есть, но их никто понятно не объяснил;
- интернет-советы общие и не про конкретного человека;
- непонятно, какие анализы сдавать;
- добавки покупаются хаотично;
- клиент боится навредить себе;
- нужно понять, где врач, где питание, где режим, где добавки.

## 6. Юридические и медицинские границы

Бот и проект не должны:

- ставить диагнозы;
- назначать лечение;
- назначать лекарственные препараты;
- заменять врача;
- выдавать медицинское заключение;
- обещать лечение заболеваний добавками;
- делать уверенные выводы из плохого фото, нечитаемого PDF или сомнительных данных.

Бот должен:

- говорить только на русском языке;
- использовать формулировки гипотез и рисков, а не диагнозов;
- честно направлять к врачу при красных флагах;
- при плохом качестве файла просить прислать документ в лучшем качестве;
- если показатель не распознан уверенно, не использовать его как факт;
- перед выдачей результата требовать human review;
- быть премиальным, деликатным, конкретным и не шаблонным.

## 7. Анализы и файлы

Клиент может отправлять:

- PDF из лаборатории;
- фото анализов;
- УЗИ;
- выписки;
- фотографии по жалобам;
- голосовые сообщения;
- ручной текстовый ввод показателей.

Правило безопасности:

- если файл читается уверенно, данные можно использовать как входные факты;
- если качество плохое, нужно попросить более качественный файл;
- если показатель распознан сомнительно, нужно уточнить конкретный показатель;
- нельзя придумывать значения, единицы измерения или выводы;
- нутрициологические референсы используются как ориентир, но не как основание для диагноза.

## 8. HelloDoc и лабораторный маршрут

Если у клиента нет анализов, бот должен предложить сдать их через HelloDoc по реферальным ссылкам Ольги:

- `https://hellodoc.app/s/27u6a/`
- `https://hellodoc.app/s/gdgjq/`

Сценарий:

- “Если анализы уже есть, пришлите PDF/фото/текст сюда.”
- “Если анализов нет, можно сдать через HelloDoc, а потом прислать результаты в бот.”

Важно: бот не обязан хранить полный каталог Helix внутри себя как лабораторную витрину. Его задача — объяснить, какие группы анализов нужны по гипотезам, а клиент выбирает и сдаёт через HelloDoc/лабораторию.

## 9. Нутрицевтики

Разрешённый формат:

- ориентиры по добавкам, если они уместны;
- форма, время приёма, совместимость, осторожность;
- приоритет: Сибирское здоровье;
- альтернатива: Vitamax;
- ссылка/промокод Ольги могут даваться клиенту даже до сдачи анализов, но без обещания лечения и без агрессивного навязывания.

Запрещённый формат:

- “лечим анемию/щитовидку/ЖКТ добавкой”;
- “это заменит врача/лекарства”;
- “точно причина в этом”;
- схемы при красных флагах без врача;
- уверенные схемы при недостатке данных.

## 10. Оплата

Текущий официальный режим пилота: `PAYMENT_MODE=manual`.

Причина: YooKassa/Telegram payment flow был нестабилен и создавал риск срыва клиентского пути.

Как работает ручной режим:

1. Клиент выбирает продукт.
2. Бот фиксирует продукт и сумму.
3. Клиент получает инструкцию по ручной оплате.
4. Оператор вручную проверяет оплату.
5. Оператор подтверждает оплату в админском сценарии.
6. Только после подтверждения запускается экспертный процесс.

YooKassa не является текущим блокером пилота, потому что ручная оплата уже принята как рабочий пилотный режим. YooKassa нужна позже для масштабирования и автоматизации.

## 11. Архитектура проекта

Текущий backend находится локально в `WellnessBot`.

Ключевые модули:

- `WellnessBot/main.py` — основной Telegram runtime, сценарии, обработчики, админские команды;
- `WellnessBot/texts.py` — клиентские тексты, меню, продуктовая витрина, пример результата;
- `WellnessBot/payment_flow.py` — продукты, суммы, платёжный контекст;
- `WellnessBot/ai_drafting.py` — генерация досье, внутренний критик, growth-аудит;
- `WellnessBot/prompts.py` — системные правила, безопасность, тон, границы;
- `WellnessBot/lab_ocr.py` — обработка лабораторных файлов;
- `WellnessBot/voice_processor.py` — обработка голосовых;
- `WellnessBot/medical_skill_database.py` — внутренняя база навыков/ориентиров;
- `WellnessBot/supplement_product_catalog.py` — каталог добавок;
- `WellnessBot/nutrition_reference_ranges.py` — нутрициологические ориентиры;
- `ops/` — запуск, перезапуск, тесты, вспомогательные сценарии;
- `docs/` — стратегия, аудит, безопасность, запуск, продуктовые решения.

Текущий запуск:

- локально на компьютере;
- не production 24/7;
- для пилота допустимо, если компьютер включён и контролируется;
- для стабильной 24/7 эксплуатации позже нужен VPS или другой production-хостинг.

## 12. AI и модели

Используется связка:

- основной Telegram-бот и локальный backend;
- DeepSeek подключён к Antigravity как аудитор/советник;
- внутренний judge критикует досье перед выдачей;
- growth architect оценивает коммерческую ценность и продуктовые слабые места;
- финальная клиентская выдача не должна уходить без human review.

DeepSeek/другая модель не является юридически или медицински ответственным финальным лицом. Она помогает анализировать, но решение о выдаче клиенту проходит через человека.

## 13. Хранение данных и приватность

Нельзя отправлять в GitHub:

- `.env`;
- API-ключи;
- Telegram token;
- DeepSeek token;
- YooKassa token;
- клиентские анализы;
- PDF/фото/выписки клиентов;
- `WellnessBot/data`;
- runtime-файлы с Telegram ID;
- приватные отчёты по конкретным клиентам;
- сырые OCR/ASR-артефакты, если там есть персональные данные.

В GitHub можно хранить:

- архитектуру;
- стратегию;
- обезличенные статусы;
- правила безопасности;
- продуктовую линейку;
- чек-листы;
- код без секретов;
- шаблоны документов без реальных паспортных/платёжных данных.

## 14. Текущие блокеры

Критические:

1. Нужно освободить диск `C:` выше безопасного порога `10 GB`, сейчас риск деградации среды высокий.
2. Есть расхождение runtime/storage по одному `week`-сценарию: runtime существует, но persisted submission JSON отсутствует.
3. У одного пользователя были параллельные ветки `week` и `premium`, что создаёт риск “двух правд”.
4. Свежая premium-ветка с плохими/неподтверждёнными анализами должна оставаться замороженной до читаемых данных.
5. Роутер live-ответов слишком много перехватывает шаблонами: часть симптомных запросов не доходит до модели.

Некритические, но важные:

- production VPS ещё не финализирован;
- YooKassa пока отложена;
- нужно собрать 1-3 реальных пилотных цикла;
- нужно доказать, что результат не выглядит шаблонным;
- нужно проверить, сколько времени реально занимает один клиент.

## 15. Ближайший план выполнения

Шаг 1. Безопасная стабилизация среды.

- Освободить диск `C:` выше `10 GB`.
- Не генерировать тяжёлые PDF/пакеты, пока диск в критическом состоянии.

Шаг 2. Привести клиентские ветки к одной правде.

- Найти активный paid path.
- Зафиксировать одну активную ветку на пользователя.
- Остальные заморозить, архивировать или слить.

Шаг 3. Усилить live-качество.

- Сузить deterministic router.
- Оставить шаблоны только для emergency/crisis/upload/logistics FAQ.
- Симптомные запросы отдавать модели с правилами безопасности.
- Добавить уточняющие вопросы там, где данных мало.

Шаг 4. Провести полный тестовый walkthrough.

- `/start`;
- выбор продукта;
- пример результата;
- анкета;
- анализы есть/нет;
- HelloDoc;
- ручная оплата;
- админское подтверждение;
- генерация черновика;
- judge;
- human review;
- клиентская выдача.

Шаг 5. Запустить 1-3 controlled pilot клиента.

- Без массовой рекламы.
- Каждое досье только после проверки.
- После каждого кейса фиксировать, где клиент запутался, где текст был общим, где нужна ручная корректировка.

## 16. Критерии pilot-ready

Проект можно считать готовым к controlled pilot, если:

- бот стабильно стартует;
- интерфейс только на русском;
- продуктовая витрина понятна;
- пример результата показывается;
- анкета проходит без зависания;
- файлы принимаются;
- при плохом качестве файлов бот просит переслать лучше;
- ручная оплата работает организационно;
- админ видит продукт, сумму, заявку;
- AI-черновик не уходит клиенту без проверки;
- нет диагнозов и лечения;
- есть понятный план клиенту;
- есть процесс поддержки после выдачи.

## 17. Критерии public launch

Публичный запуск нельзя делать, пока не выполнено:

- минимум 3-5 реальных клиентских циклов;
- нет критических сбоев в анкете, файлах, оплате, досье и админке;
- human review работает стабильно;
- понятен порядок хранения данных;
- понятен юридически аккуратный способ оплаты;
- есть отзывы и подтверждённая ценность;
- понятно, сколько времени требует один клиент;
- принято решение по VPS/24-7 работе.
