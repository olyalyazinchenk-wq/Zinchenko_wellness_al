# Task Selection Policy

## Приоритеты
1. **P0:** блокирует pilot/public launch → немедленно.
2. **P1:** важное, до первого клиента → сегодня/завтра.
3. **P2:** значимое улучшение → на этой неделе.
4. **P3:** технический долг/nice-to-have → когда есть окно.

## Критерии выбора задачи
- Снимает ли конкретный риск?
- Можно ли сделать быстро (≤ 2 часов)?
- Не требует ли forbidden actions?
- Есть ли approved packet?

## Что НЕ брать
- Задачи без явного approval, требующие кода.
- Задачи с размытым scope («улучшить продукт»).
- Задачи, где риск изменения > риск проблемы.
- Повторные аудиты без новых данных.

## Правило «одна задача — один результат»
Не смешивать аудит и реализацию в одной задаче.
Сначала аудит → draft packets → approval → реализация.
