# Стратегия controlled pilot v2: Olga Wellness AI

Дата: 2026-05-05
Статус: рабочий стратегический план после runtime-проверки, smoke-тестов и аудита Hermes
Владелец: Codex + Olga
Режим проекта: controlled concierge pilot, не public launch

## 1. Главный вывод

Проект уже не находится в стадии “идея/черновик”. У нас есть работающий Telegram backend, подключенный DeepSeek, ручная оплата, smoke-тесты payment/admin/dossier и система параллельного агента Hermes.

Но проект еще не готов к публичному запуску. Лучшее решение сейчас: не расширять поверхность продукта, а довести один безопасный платный путь до состояния pilot-ready.

Целевой путь:

`/start -> пример результата -> выбор продукта -> анкета -> анализы/нет анализов -> ручная оплата -> admin confirm -> AI draft -> judge -> human review -> delivery -> сопровождение`

## 2. Что уже работает

- Telegram-бот запущен и обрабатывает сообщения.
- DeepSeek отвечает через OpenAI-compatible API.
- Payment smoke прошел: ручная оплата и валидация суммы работают на тесте.
- Admin governance smoke прошел.
- Complex dossier smoke прошел: JSON, judge report и PDF создаются.
- Добавлен LLM timeout, чтобы генерация не зависала бесконечно.
- Hermes подключен как Level 1 агент: отчеты, backlog, task packets, навыки.
- Создана база контекста для новых моделей.
- Домен/сайт/mini-app стратегически разведены: сначала landing/legal, потом безопасный mini-app.

## 3. Главная стратегическая коррекция

Старый риск “runtime unavailable” частично снят: бот сейчас работает.

Новый live-риск: OCR Яндекса возвращает `401 Unauthorized` при обработке фото/файлов. Это означает, что часть клиентских анализов может не распознаваться автоматически. Для premium-продукта это критично, потому что клиент ожидает корректную работу с анализами.

Решение:

- не обещать автоматическое распознавание как гарантированный результат;
- в коде и текстах иметь понятный fallback: “файл не удалось надежно распознать, пришлите PDF/лучшее фото или введите показатели текстом”;
- проверить STT/OCR ключи отдельно, без раскрытия секретов;
- до первого пилота убедиться, что плохие/нечитаемые файлы не становятся источником выдуманных данных.

## 4. P0 до первого платного пилота

### P0.1 Delivery gate

Проблема: клиент не должен получить досье, если judge verdict требует доработки.

Решение:

- добавить guard перед delivery;
- если `needs_revision`, `must_rewrite`, `reject` - доставка блокируется;
- manual override возможен только с note/by/timestamp;
- smoke-тест должен подтвердить блокировку.

### P0.2 Canonical client path

Проблема: один клиент может иметь несколько активных веток и состояний.

Решение:

- определить одну активную заявку на пользователя/продукт;
- старые ветки помечать archived/stale;
- новые документы после оплаты прикреплять к активному кейсу, а не создавать хаос.

### P0.3 OCR/file fallback

Проблема: OCR Яндекса дает 401, значит автоматическое чтение фото нестабильно.

Решение:

- проверить переменные STT/OCR без вывода ключей;
- добавить/проверить мягкий fallback для клиента;
- запретить использование неуверенно распознанных данных как фактов;
- live-тест: PDF, хорошее фото, плохое фото, ручной ввод.

### P0.4 Mini-app/public surface drift

Проблема: UI-прототипы/mini-app могут показывать старые цены или медицински рискованные placeholder findings.

Решение:

- пока не делать mini-app публичной поверхностью;
- использовать домен сначала как landing/legal/site;
- любые UI-прототипы маркировать как reference/mock;
- привести цены и формулировки к официальной линейке перед публикацией.

## 5. P1 до первого реального клиента

- Русский язык в интерфейсе: убрать английские фразы вроде `Premium Wellness Dossier`.
- Синхронизация цен: 0 / 3900 / 6900 / 14900 везде.
- Ручная оплата: понятная инструкция, контакт оператора, статус “ожидаем подтверждение”.
- Prompt hardening: весь клиентский результат только на русском, без диагностики/лечения.
- Sanitizer live replies: расширить защиту от уверенных медицинских формулировок.
- Consent: фиксировать согласие, время, контекст, ограничения сервиса.
- Возраст/особые группы: дети, подростки, беременность/ГВ, онкология - осторожность и врач.
- Admin review: админ должен видеть топ-риски из judge, а не просто “надо проверить”.

## 6. P2 после первого стабильного пилота

- Сократить демо-пример и поставить кнопку “Посмотреть пример” выше.
- Добавить “быструю победу” после анкеты: 1-2 безопасных наблюдения без выдачи диагноза.
- Добавить “почему Ольга”: короткое доверительное позиционирование.
- Подготовить сайт на домене: тарифы, пример результата, privacy, consent, кнопка в Telegram.
- Подготовить post-delivery сопровождение на 30 дней.
- Подготовить dashboard/admin improvements после стабилизации основного пути.

## 7. Что НЕ делаем сейчас

- Не запускаем публичный поток.
- Не переносим весь проект на mini-app до стабилизации backend flow.
- Не строим сложную CRM до первого пилота.
- Не автоматизируем YooKassa, пока ручной платеж достаточен для пилота.
- Не даем AI напрямую клиенту без human review.
- Не расширяем каталог анализов до бесконечности, пока не стабилен базовый file/lab flow.

## 8. Роли

### Codex

- кодовые исправления P0/P1;
- runtime, smoke, tests;
- GitHub source of truth;
- интеграция live-test feedback;
- выпуск pilot-ready checklist.

### Hermes

- Level 1: аудит, backlog, draft task packets, skill improvements;
- может готовить copy/safety/UX предложения;
- не меняет код без отдельного approved implementation packet.

### Olga

- проводит живой тест как клиент;
- присылает скрины/тексты/что не понравилось;
- дает домен и публичные реквизиты без паролей;
- утверждает короткий профиль “почему Ольга”;
- утверждает юридические тексты и границы продукта.

## 9. Как обрабатываем живые тесты Ольги

Каждый live feedback превращаем в карточку:

- ID;
- шаг пути;
- что увидела Ольга;
- ожидаемое поведение;
- фактическое поведение;
- риск: P0/P1/P2/P3;
- тип: UX / safety / payment / lab / runtime / copy / legal / growth;
- решение;
- исполнитель;
- проверка.

Приоритет:

1. Safety/payment/delivery blockers.
2. Невозможность пройти путь.
3. Ошибки анализов/файлов/OCR.
4. Нерусский/некрасивый/непремиальный текст.
5. Коммерческие улучшения.

## 10. План на ближайшие 72 часа

### День 1: закрыть P0-путь

1. Разобрать live test Ольги.
2. Исправить delivery gate.
3. Проверить OCR/STT/fallback на файлах.
4. Зафиксировать canonical active case logic или временный ручной регламент.
5. Перепроверить bot-status + payment/admin/dossier smoke.

### День 2: сделать клиентский путь премиальным и понятным

1. Убрать английский/битые/старые тексты.
2. Синхронизировать цены и продукты.
3. Усилить ручную оплату и связь с оператором.
4. Усилить demo/next action.
5. Подготовить admin review text: топ-3 риска из judge.

### День 3: pilot-ready gate

1. Полный live e2e walkthrough: intake -> files -> payment -> draft -> judge -> admin review -> delivery.
2. Pilot launch checklist.
3. Решение: pilot-ready / pilot-ready with conditions / not ready.
4. Если ready - запускаем 1 controlled concierge клиента вручную.

## 11. План на 7 дней

1. Дни 1-3: P0/P1 стабилизация Telegram backend.
2. День 4: сайт/домен minimum viable landing + legal pages draft.
3. День 5: улучшение premium output и 30-day support flow.
4. День 6: повторный smoke + live rehearsal.
5. День 7: первый controlled concierge клиент или финальный pre-pilot исправительный день.

## 12. Критерии pilot-ready

Проект можно запускать на первого controlled concierge клиента, если:

- бот стабильно отвечает;
- клиент может пройти /start -> анкета -> файлы/нет файлов -> оплата;
- ручная оплата понятна;
- клиент не получает досье без admin/human review;
- `needs_revision` не уходит клиенту без manual override;
- плохие файлы не превращаются в выдуманные показатели;
- цены везде совпадают;
- интерфейс на русском;
- клиент понимает, что это не медицинская диагностика;
- есть контакт оператора;
- есть план действий при сбое.

## 13. Следующий рабочий шаг

Ждем live-тесты Ольги.

Как только Ольга присылает скрин/текст/описание сбоя, Codex делает:

1. triage;
2. присваивает priority;
3. связывает с текущим P0/P1 backlog;
4. чинит P0/P1;
5. запускает smoke;
6. фиксирует результат.

Главная линия: меньше расширения, больше доведения одного безопасного платного пути до результата.
