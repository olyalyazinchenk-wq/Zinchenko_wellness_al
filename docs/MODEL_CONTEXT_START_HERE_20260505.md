# Model Context Start Here: Olga Wellness AI

Дата: 2026-05-05
Статус: canonical sanitized onboarding packet
Назначение: быстрый вход в проект для Codex, Hermes, Antigravity, DeepSeek, Gemini/GitHub-моделей и других доверенных помощников.

## 1. Решение

Для проекта вводится многоуровневая база памяти:

- GitHub `docs/` - главный безопасный source of truth для моделей, разработчиков и аудиторов.
- Локальная папка `docs/` - рабочая копия и оперативная память на компьютере Ольги.
- `docs/obsidian_mirror/` - офлайн-зеркало для Obsidian/ручного чтения.
- Notion - человеческая панель навигации: статусы, решения, ссылки, задачи, без секретов и без клиентских данных.
- `.env`, локальные секрет-хранилища и настройки рантайма - только для ключей, токенов и приватных доступов.
- `WellnessBot/data/`, клиентские PDF/фото/голосовые/выписки - приватные runtime-данные, не являются общей базой для моделей.

Главная идея: любая новая модель должна читать не весь хаос проекта, а короткий входной маршрут и затем выполнять конкретный task packet.

## 2. Что мы строим

Telegram-first сервис нутрициологической навигации Ольги Зинченко:

`анкета -> ручная оплата -> сбор файлов -> AI-черновик -> внутренний критик -> human review -> клиентское досье -> сопровождение`

Это не медицинский диагностический бот и не сервис назначения лечения. Это премиальная нутрициологическая навигация: сбор картины, гипотезы по дефицитным рискам и перегрузкам, безопасные следующие шаги, вопросы к врачу, план режима/питания/нутрицевтических ориентиров.

## 3. Текущий этап

Этап: controlled concierge pilot.

Публичный масштабный запуск пока преждевременен. Главная цель ближайшего этапа - довести один платный путь до стабильности:

1. Клиент понимает продукт и видит пример результата.
2. Клиент проходит расширенную анкету.
3. Клиент присылает анализы/файлы или получает маршрут через HelloDoc.
4. Оплата подтверждается вручную.
5. Бот готовит черновик.
6. Черновик проходит safety/judge review.
7. Ольга проверяет и только затем отправляет клиенту.
8. В рамках тарифа клиент может досылать вопросы и документы.

## 4. Где читать контекст новой модели

Минимальный порядок чтения:

1. `docs/PROJECT_DEVELOPMENT_SINGLE_SOURCE_OF_TRUTH.md`
2. `docs/AGENT_CONTEXT_HUB.md`
3. `docs/PROJECT_PULSE_LOG.md` - последние записи, не обязательно весь файл.
4. `docs/DOMAIN_SITE_MINIAPP_STRATEGY_20260505.md`
5. `docs/PRODUCT_LINE_V2_20260426.md`
6. `docs/NUTRITION_NAVIGATION_POLICY_20260420.md`
7. `docs/LAB_RESULT_SAFETY_POLICY_20260421.md`
8. `docs/MODEL_CONTEXT_START_HERE_20260505.md` - этот документ.
9. `docs/HERMES_PROJECT_WORKER_PROTOCOL_20260505.md`, если работает Hermes или другой локальный агент.
10. `docs/templates/HERMES_TASK_PACKET_TEMPLATE_20260505.md`, если нужно выдать конкретную задачу агенту.

Правило: если документы противоречат друг другу, приоритет имеет более свежий документ, но медицинско-юридические ограничения имеют приоритет всегда.

## 5. Что можно давать моделям

Можно давать:

- стратегию, продуктовую линейку, архитектуру, планы;
- обезличенные тест-кейсы;
- шаблоны промптов и safety-правила;
- код без секретов;
- список задач и критерии проверки;
- ссылки на публичные репозитории и публичные страницы.

Нельзя давать в общую базу:

- `.env`;
- API-ключи DeepSeek, Telegram, YooKassa, GitHub, Beget, VPS;
- реальные клиентские анализы, фото, PDF, выписки, голоса;
- номера телефонов, Telegram ID, ФИО клиентов;
- платежные подтверждения;
- любые персональные данные без явной необходимости и обезличивания.

## 6. Роли моделей

Codex:

- архитектор и исполнитель проекта;
- код, документы, GitHub, проверка, запуск, стратегия;
- ведет single source of truth.

Hermes:

- локальный доверенный рабочий агент;
- может читать sanitized context pack;
- выполняет только утвержденные task packets;
- пишет отчеты в понятном формате;
- не получает секреты из общей базы.

Antigravity/DeepSeek:

- сильный аудитор, критик, разработчик черновиков;
- полезен для проверки логики, промптов, UX, безопасности;
- не является финальным медицинским лицом.

Judge/Safety Auditor:

- ищет слабые места;
- блокирует выдачу при красных флагах, медицинских обещаниях, плохом качестве данных, путанице показателей.

Growth Architect:

- проверяет коммерческую ценность, позиционирование, воронку, понятность результата.

## 7. Обязательные продуктовые правила

- Интерфейс и ответы клиенту только на русском языке.
- Никакой диагностики и назначения лечения.
- Только гипотезы, риски, ориентиры и следующие шаги.
- При красных флагах - врач, не добавки.
- При плохом файле - попросить качественный файл.
- Если показатель сомнителен, не использовать его как факт.
- Нутрицевтики: Сибирское здоровье и Vitamax как ориентиры, без обещаний лечения.
- Клиент должен видеть конкретный план: 3 дня, 2 недели, 1 месяц, 1-3 месяца.
- Клиент должен понимать, что премиальный продукт отличается от бесплатных советов конкретностью, персонализацией и сопровождением.

## 8. Текущие системные решения

- Оплата на пилоте: ручная.
- YooKassa не блокирует пилот, но нужна позже для автоматизации.
- Домен нужен: сначала сайт/витрина/юридические страницы, потом безопасный Telegram Mini App.
- `moy-projekt` из Google AI Studio - UI-прототип/референс админки и клиентского flow, не production backend.
- Локальный бот - текущий реальный backend.
- VPS понадобится для 24/7 production, но пилот можно проверять локально.

## 9. Как агент должен работать

Перед началом работы агент обязан:

1. Прочитать этот документ и документы из раздела 4.
2. Проверить свежий `git status`.
3. Уточнить, есть ли approved task packet.
4. Не трогать секреты и клиентские данные.
5. Не деплоить и не менять оплату без отдельного разрешения.
6. Выполнять задачу маленькими проверяемыми шагами.
7. В конце оставить отчет: что сделано, какие файлы изменены, как проверено, какие риски остались.

## 10. Ближайшая траектория проекта

1. Зафиксировать базу памяти и протокол Гермеса.

6. Использовать `moy-projekt` как дизайн-референс для будущего Mini App/admin, а не как немедленную замену backend.
