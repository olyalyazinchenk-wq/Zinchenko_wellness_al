# Global Execution Master Plan — Olga Wellness AI

Дата: 2026-05-06
Статус: active execution plan
Режим: controlled concierge pilot -> first paid client -> repeatable sales system

## 1. North Star

Цель проекта — не “бот ради интереса”, а продаваемый Telegram-first сервис нутрициологической навигации Ольги Зинченко.

Итоговая система должна выглядеть так:

`маркетинг -> Telegram bot -> пример результата -> продукт -> анкета -> файлы/анализы -> ручная оплата -> AI draft -> judge -> human review -> клиентское досье -> 30-дневное сопровождение -> отзыв/повторная продажа`

Ключевой принцип: AI помогает собирать, структурировать и подсвечивать гипотезы, но не ставит диагнозы, не назначает лечение и не отправляет клиенту результат без human review.

## 2. Текущая фактическая точка

Уже есть:

- Telegram-бот работает и отвечает.
- DeepSeek подключен и отвечает.
- Ручная оплата работает в smoke-тесте.
- Admin/governance smoke работает.
- Сложное досье генерируется, judge/report/PDF создаются.
- Delivery gate внедрен: рискованное досье не должно уйти клиенту без override.
- Invoice payload исправлен для product codes `week/premium/vip`.
- Клиентский CTA частично русифицирован.
- Расширены безопасные biomarker aliases.
- Hermes работает параллельно как Level 1 штаб: отчеты, task packets, playbooks.

Остается:

- OCR 401 / file-lab reliability.
- Canonical client path: один клиент -> один активный кейс.
- Проверка live-тестов Ольги.
- Доведение клиентских текстов до премиального русского UX.
- Pilot readiness checklist.
- Landing/legal/payment trust layer.
- Маркетинговая упаковка и монетизация.

## 3. Главный стратегический выбор

Мы НЕ запускаем публичный поток сейчас.

Мы запускаем controlled concierge pilot:

1. 1-3 ручных клиента.
2. Все оплаты вручную.
3. Все досье через human review.
4. Все сбои фиксируются как backlog.
5. После 1-3 успешных клиентов — решение о public launch / VPS / mini-app / YooKassa automation.

Это лучший путь, потому что он быстрее приносит реальные деньги и реальные данные, но не подвергает проект репутационному и медицинско-юридическому риску.

## 5. Monetization Strategy

### Сразу

- ручная оплата;
- Telegram как основной канал;
- продажа через доверие, пример результата и понятный next action;
- не ждём YooKassa/mini-app для первых денег.

### После 1-3 пилотов

- лендинг на домене;
- privacy/consent/legal pages;
- отзывы;
- кейсы без персональных данных;
- Telegram-контент с pain triggers;
- простая воронка: пост -> бот -> demo -> платный разбор.

### После стабильности

- YooKassa/Telegram payments;
- mini-app как аккуратный интерфейс, а не замена backend;
- VPS для 24/7;
- dashboard/admin;
- партнерские ссылки: HelloDoc, Сибирское здоровье, Vitamax.

## 6. Marketing Policy

Позиционирование:

`Не диагноз и не лечение. Премиальная нутрициологическая навигация: собрать картину, увидеть риски, понять следующий безопасный шаг и подготовиться к врачу/коррекции образа жизни.`

Что продаем:

- ясность;
- структурный взгляд на кейс;
- безопасный маршрут;
- человеческую поддержку;
- план действий;
- экономию времени и хаоса;
- аккуратную работу с анализами;
- не “волшебную таблетку”.

Запрещено в маркетинге:

- обещать лечение;
- обещать точную диагностику;
- обещать “100% распознавание анализов”;
- обещать, что добавки заменят врача/лекарства;
- продавать страхом без маршрута помощи.

## 7. P0 Execution Track

### P0.1 File/Lab Flow Reliability

Цель: клиентские файлы не должны превращаться в хаос.

Действия:

1. OCR preflight без вывода секретов.
2. Проверить `401 Unauthorized`.
3. Убедиться, что PDF extraction работает без OCR, если PDF содержит текст.
4. Проверить fallback для плохого фото/OCR error.
5. Live cases: PDF, хорошее фото, плохое фото, ручной ввод.

### P0.2 Canonical Client Path

Цель: один клиент не должен иметь несколько активных веток.

Действия:

1. Найти точки создания submission.
2. Добавить/утвердить active-case policy.
3. Документы после оплаты/доставки прикреплять как follow-up.
4. Старые ветки архивировать/не показывать как активные.

### P0.3 Delivery/Human Review Gate

Статус: первый фикс внедрен.

Дальше:

1. Добавить smoke на блокировку delivery.
2. Добавить manual override audit trail, если нужен.
3. Проверить live admin approval.

### P0.4 Live Test Triage

Цель: каждый скрин Ольги -> action item.

Действия:

1. Triage: шаг, ожидание, факт, риск, fix, owner, verify.
2. P0/P1 чинить сразу.
3. P2/P3 в backlog.

## 11. Work Rhythm

Каждый цикл работы:

1. Проверить bot status.
2. Взять самый высокий P0/P1.
3. Сделать минимальный безопасный fix.
4. Проверить smoke/unit/live.
5. Коммит в GitHub.
6. Обновить decision/log только если это реально меняет стратегию.
7. Вернуться к next P0/P1.

## 12. Критерии “готовы к первому платному клиенту”

- Бот стабильно отвечает.
- Клиент проходит путь до оплаты.
- Файлы/анализы имеют безопасный fallback.
- Ручная оплата понятна.
- AI draft создается.
- Judge работает.
- Human review обязателен.
- Рискованное досье не доставляется.
- Цены совпадают.
- Интерфейс русский.
- Есть операторский контакт.
- Есть план при сбое.
