# Canonical Client Path Audit

## Где создаётся новая заявка
- `main.py` → intake flow → `build_initial_submission_payload` → новый `submission_id`.
- При `/start` или «хочу разбор» → создаётся новая сессия.

## Где восстанавливается сессия
- `case_service.py` → `build_session_from_submission` — восстанавливает из сохранённого JSON.
- Если `submission_id` передан → загружается конкретный кейс.

## Где выбирается active case
- **Проблема:** нет enforcement «один активный кейс на пользователя».
- Код проверяет наличие сессии, но не блокирует создание параллельных веток.
- `build_submission_id` создаёт новый ID без проверки существующих активных.

## Где follow-up привязывается к кейсу
- `main.py:587` — проверка `intake_status == "delivered_to_client"` → follow-up mode.
- Если статус ≠ delivered, сообщения могут создать новую ветку.

## Риски multi-path

| Риск | Вероятность | Последствия |
|------|------------|-------------|
| Клиент начинает новый /start, не завершив старый | Высокая | 2+ активные сессии |
| Клиент присылает документы после доставки → новый кейс | Средняя | Разрыв follow-up |
| Оператор путает ветки → правит не тот кейс | Средняя | Неверная доставка |
| Новый intake поверх существующего | Низкая | Потеря данных |

## Временный ручной регламент (до кодового фикса)

1. **Оператор перед подтверждением оплаты проверяет:** есть ли другие активные submission у этого пользователя.
2. **Если есть → выбрать delivered как канонический.** Новые документы в follow-up.
3. **Если несколько pending → заморозить старые,** работать с последним.
4. **Не создавать новый intake,** пока предыдущий не delivered.

## Минимальный кодовый фикс (draft для Codex)

1. При создании нового intake → проверить `list_recent_cases` для user_id.
2. Если есть delivered → переключить в follow-up mode (не создавать новый submission).
3. Если есть pending → спросить: «У вас уже есть активная заявка. Продолжить её или начать новую?»
4. Старые submission помечать `archived` при создании нового.
