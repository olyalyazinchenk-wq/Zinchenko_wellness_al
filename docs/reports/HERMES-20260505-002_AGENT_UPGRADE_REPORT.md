## 1. Что я понял о проекте

**Продукт:** Olga Wellness AI — Telegram-first сервис премиальной нутрициологической навигации Ольги Зинченко.

**Целевой цикл:**
`анкета → ручная оплата → сбор файлов/анализов → AI-черновик → внутренний критик → human review → клиентское досье → сопровождение`

**Текущий этап:** controlled concierge pilot. Публичный запуск заблокирован. Задача — довести один платный клиентский цикл до стабильности.

**Продуктовая линейка (принято):**
- Демо-пример: 0 ₽
- Разбор на 7 дней: 3 900 ₽
- Персональный разбор на 30 дней: 6 900 ₽
- VIP-сопровождение на 30 дней: 14 900 ₽

**Ключевые блокеры (P0):**
1. Delivery-gate bypass: кейс доставлен клиенту при вердикте `needs_revision`.
2. Same-user multi-path drift: 1 пользователь = 4 активные ветки (1 week + 2 premium + 1 stale).
3. Mini-app truth/pricing drift: показывает 2990 ₽ вместо официальных цен, хардкодит медицинские findings.
4. Runtime resilience regression: бот не evidenced как running, proxy failures на `127.0.0.1:12334`.

**Архитектура:**
- `WellnessBot/` — основной backend (Python + aiogram)
- `WellnessTMA/` — Telegram Mini App (React/Vite)
- `moy-projekt` (Google AI Studio) — UI-прототип, НЕ production backend
- `nutrition_bot` (WSL) — архитектурный прототип Hermes с сильными AI-промптами, подлежит архивации

**Границы безопасности:**
- Не диагноз, не лечение, не медицинское заключение.
- Формулировки — только гипотезы и ориентиры.
- Human review обязателен перед выдачей клиенту.
- Красные флаги → врач.
- Плохое качество анализов → запрос переслать, а не гадание.

## 4. Как работает мой цикл самоулучшения

После каждой задачи — обязательный ритуал (навык #10):

1. **Анализ повторяемости:** что делал не в первый раз?
2. **Кандидат в навык:** можно ли превратить в переиспользуемый навык?
3. **Анализ ошибок:** что пошло не так и как предотвратить?
4. **Проверка существующих навыков:** нужно ли обновить?
5. **Запись в отчёт:** секция «Выводы для самоулучшения».
6. **Обновление индекса:** новый навык → в `index.md`.

**Целевые метрики:**
- После 3 задач — не менее 1 нового/обновлённого навыка.
- После 5 задач — заметное ускорение входа в контекст.
- Ни одного повторения одной и той же ошибки.

**Важно:** «самообучение» = внешняя рабочая память (навыки, шаблоны, чек-листы), а не изменение весов модели.

## 5. Лестница автономности

### Level 0: Read-only аудит
**Можно:** читать `docs/`, `WellnessBot/` (архитектура), `ops/`, `README.md`.
**Нельзя:** менять любые файлы.
**Отчёт:** наблюдения и рекомендации.
**Approval:** не требуется для чтения.

### Level 1: Документация и task packets
**Можно:** создавать/редактировать `docs/hermes_skills/`, `docs/reports/`, `docs/tasks/HERMES-*.md`.
**Нельзя:** менять код, `.env`, клиентские данные.
**Отчёт:** полный отчёт по форме task packet.
**Approval:** task packet должен быть `approved`.

### Level 2: Тексты, UX, тест-кейсы
**Можно:** править тексты в `WellnessBot/texts.py` (по задаче), создавать тест-кейсы, проверять UX.
**Нельзя:** менять логику бота, оплату, выдачу.
**Отчёт:** diff + safety-аудит изменений.
**Approval:** Codex review перед применением.

### Level 3: Ограниченные правки кода
**Можно:** точечные правки в `WellnessBot/*.py` по конкретному task packet.
**Нельзя:** менять архитектуру, `main.py` (без review), payment flow, production-конфиги.
**Отчёт:** diff + прогон smoke-тестов (если доступны).
**Approval:** Codex/Olga approval + human review изменений.

### Level 4: Запуск проверок/смоков
**Можно:** запускать `ops/` скрипты, тесты, проверки качества.
**Нельзя:** деплоить, менять production, запускать массовые сценарии.
**Отчёт:** результаты прогона + регрессии.
**Approval:** для запуска на production-окружении — отдельное approval.

### Level 5: Production/VPS/Payment/Client-facing
**Можно:** только после отдельного письменного approval на каждое действие.
**Нельзя:** всё без approval.
**Отчёт:** детальный лог действий с подтверждением approval.
**Approval:** обязателен всегда.
