# Telegram Bot Flow Auditor

## Когда использовать
При проверке клиентского пути в Telegram-боте, поиске дыр в сценариях, проверке UX.

## Входные данные
- Файлы бота: `WellnessBot/main.py`, `WellnessBot/texts.py`, `WellnessBot/payment_flow.py`.
- Продуктовая линейка: `docs/PRODUCT_LINE_V2_20260426.md`.
- Правила оплаты: `docs/MANUAL_PAYMENT_MODE_20260426.md`.

## Пошаговый workflow

1. **Построить карту потока:**
   - `/start` → продуктовая витрина → демо / выбор продукта.
   - Анкета → сбор файлов → HelloDoc (если нет анализов).
   - Оплата (ручная) → подтверждение админом.
   - Генерация черновика → judge → human review → выдача клиенту.
   - Follow-up в течение периода продукта.

2. **Найти разрывы:**
   - Можно ли получить результат до оплаты?
   - Можно ли получить результат без human review?
   - Можно ли обойти анкету?
   - Есть ли места, где бот зависает без ответа?

3. **Проверить delivery gate:**
   - Блокируется ли выдача при `needs_revision`?
   - Есть ли явный manual override для исключений?

4. **Проверить multi-path:**
   - Может ли один пользователь иметь несколько активных веток?
   - Какая ветка считается канонической?

5. **Проверить меню и кнопки:**
   - Все ли кнопки ведут туда, куда должны?
   - Нет ли мёртвых кнопок?
   - Отображаются ли актуальные цены (3900 / 6900 / 14900)?

## Критерии качества
- Карта потока полная.
- Найденные дыры зафиксированы с уровнем опасности.
- Предложены конкретные места правки (файл + строка/функция).

## Что запрещено
- Менять код бота в рамках этой задачи (если не разрешено task packet).
- Проверять на реальных клиентских данных.
- Менять оплату / выдачу / production.

## Формат результата
1. Карта потока (текстовая схема).
2. Таблица дыр: Локация | Проблема | Уровень | Рекомендация.
3. Статус delivery gate.
