# Lab Data Safety Reviewer

## Когда использовать
При проверке работы с лабораторными данными: OCR, интерпретация, использование в досье.

## Входные данные
- Код: `WellnessBot/lab_ocr.py`, `WellnessBot/ai_drafting.py` (части про анализы).
- Политики: `docs/LAB_RESULT_SAFETY_POLICY_20260421.md`, `docs/NUTRITION_NAVIGATION_POLICY_20260420.md`.
- Тестовые/обезличенные данные (НЕ реальные клиентские).

## Пошаговый workflow

1. **Проверить качество OCR:**
   - Обрабатывает ли система ситуацию плохого качества?
   - Есть ли проверка `requires_lab_resubmission == true`?
   - При плохом качестве — просит переслать, а не гадает?

2. **Проверить интерпретацию:**
   - Используются ли значения как факты только при уверенном распознавании?
   - Не привязывается ли весь вывод к лабораторным референсам?
   - Интерпретация через нутрициологические ориентиры, а не медицинские диагнозы?

3. **Проверить цепочку:**
   - Распознавание → гипотеза → НЕ схема лечения.
   - При красных флагах — врач, а не добавки.
   - Сомнительный показатель не используется как основание для вывода.

4. **Проверить хранение:**
   - Не экспортируются ли реальные анализы в GitHub/Notion?
   - Локальное хранение соответствует PD-архитектуре?

## Критерии качества
- Все safety gate для плохих файлов на месте.
- Нет путей, где сомнительные данные трактуются как факты.
- Цепочка «данные → гипотеза → ориентир» не нарушена.

## Что запрещено
- Работать с реальными клиентскими анализами без явного разрешения.
- Менять safety-правила в сторону ослабления.
- Добавлять автоматическую выдачу без human review.

## Формат результата
1. Статус каждого safety gate (pass / fail / not applicable).
2. Найденные риски с указанием файла и строки.
3. Рекомендации по усилению.
