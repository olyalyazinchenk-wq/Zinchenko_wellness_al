# Wellness Safety Auditor

## Когда использовать
При проверке любых клиентских текстов, промптов, досье, ответов бота, лендингов — всего, что может быть увидено клиентом.

## Входные данные
- Текст для аудита (промпт, ответ бота, черновик досье, текст лендинга, кнопка меню).
- Политики: `docs/NUTRITION_NAVIGATION_POLICY_20260420.md`, `docs/LAB_RESULT_SAFETY_POLICY_20260421.md`.

## Пошаговый workflow

1. Просканировать текст на красные флаги:
   - Диагнозы («у вас анемия», «у вас гипотиреоз»).
   - Лечение («лечим ЖКТ», «принимайте для лечения»).
   - Замена врача («это заменит поход к врачу»).
   - Обещания лечения добавками («Сибирское здоровье вылечит»).
   - Уверенные выводы из сомнительных данных.
2. Проверить тон: премиальный, деликатный, тёплый, не шаблонный.
3. Проверить структуру: есть ли план (3 дня / 2 недели / 1 месяц / 1-3 месяца).
4. Проверить эскалацию: при красных флагах — «Это стоит обсудить с врачом».
5. Проверить human review gate: не уходит ли черновик клиенту без проверки.
6. Проверить обработку плохих файлов: бот просит переслать, а не гадает.

## Критерии качества
- Все красные флаги зафиксированы.
- Указан уровень опасности (critical / high / medium).
- Предложен безопасный переформулированный вариант.

## Pitfall: не верь статусу delivered_to_client

**Правило:** статус `delivered_to_client` в submission JSON НЕ является гарантией безопасности. Всегда проверяй `review_signals.judge_verdict`. Если вердикт содержит `needs_revision`, `must_rewrite`, `must_rewrite_with_high_caution`, `reject` — досье не должно было быть доставлено. Это известный баг (F01), требующий ручной проверки до исправления кода.

## Что запрещено
- Самому формулировать медицинские выводы при исправлении.
- Добавлять уверенность туда, где её не должно быть.
- Убирать обязательные safety gate.

## Формат результата
Таблица: Локация (файл:строка) | Проблема | Уровень | Предлагаемая замена.
